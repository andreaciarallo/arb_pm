## Summary

Phase 4 is purely additive — no changes to detection, execution, or risk logic. It layers three new capabilities on top of the working Phase 3 engine: (1) accurate fee and per-arb analytics logging to SQLite, (2) Telegram alerting for key trading events, and (3) a FastAPI web dashboard served from the same async process.

The most architecturally significant decision is running FastAPI inside the bot's existing asyncio event loop via `asyncio.create_task(uvicorn.Server(...).serve())`. This avoids subprocess complexity and keeps shared state (RiskGate, PriceCache, trade metrics) accessible from both the scan loop and the HTTP handlers without inter-process communication. The dashboard reads from SQLite and from shared in-memory state passed by reference at startup.

The Telegram integration is deliberately minimal: a single `telegram.Bot` instance initialized at startup, used only for `await bot.send_message(chat_id, text)` wrapped in try/except. No Application, no polling, no webhook — pure fire-and-forget. The daily summary is implemented as a long-lived asyncio task that computes seconds-until-midnight and sleeps until then.

**Primary recommendation:** Use FastAPI + uvicorn 0.44 background server task, python-telegram-bot 22.7 standalone Bot instance, single `src/bot/dashboard/` module with shared app-state object passed at startup.

### Pattern 1: FastAPI as asyncio Background Task

**What:** Start uvicorn programmatically using `uvicorn.Server.serve()` as an asyncio coroutine, launched with `asyncio.create_task()`. This runs the HTTP server in the same event loop as the scan loop.

**When to use:** Any time you need HTTP serving alongside a long-running asyncio scan loop without subprocess overhead.

**Key insight:** `uvicorn.Server(config).serve()` is a coroutine — it can be awaited or wrapped in `create_task`. Use `config.loop = "none"` to tell uvicorn not to create its own event loop.

```python
# Source: uvicorn programmatic usage (HIGH confidence from uvicorn docs/source)
import uvicorn
from bot.dashboard.app import create_app, AppState

async def _start_dashboard(app_state: AppState, port: int = 8080):
    app = create_app(app_state)
    config = uvicorn.Config(app, host="0.0.0.0", port=port, loop="none", log_level="warning")
    server = uvicorn.Server(config)
    await server.serve()

# In live_run.run():
dashboard_task = asyncio.create_task(_start_dashboard(app_state))
```

**Shutdown:** Cancel `dashboard_task` in the finally block alongside `ws_task`.

**When to use:** Whenever FastAPI endpoints need to read from the same in-memory state as the scan loop, without inter-process communication.

### Pattern 4: Telegram Fire-and-Forget

**What:** A `TelegramAlerter` class wrapping a `telegram.Bot` instance. All public methods are `async def` but the caller always uses `asyncio.create_task()` — the scan loop never awaits them. Failures are caught internally and logged.

**When to use:** Whenever trading logic must not block on notification delivery.

```python
# Source: python-telegram-bot v21+ Bot API (HIGH confidence — official PTB docs)
from telegram import Bot
from telegram.error import TelegramError
from loguru import logger

class TelegramAlerter:
    def __init__(self, token: str | None, chat_id: str | None):
        self._bot = Bot(token=token) if token else None
        self._chat_id = chat_id

    async def send(self, text: str) -> None:
        """Fire-and-forget send. Never raises — logs failures internally."""
        if not self._bot or not self._chat_id:
            return
        try:
            async with self._bot:
                await self._bot.send_message(
                    chat_id=self._chat_id,
                    text=text,
                    parse_mode="HTML",
                )
        except TelegramError as e:
            logger.warning(f"Telegram alert failed: {e}")
        except Exception as e:
            logger.warning(f"Telegram unexpected error: {e}")

# Caller pattern (in execute_opportunity or live_run):
asyncio.create_task(alerter.send(f"<b>Arb complete</b>\n{market_question}\nNet PnL: ${net_pnl:.4f}"))
```

**Important:** `async with self._bot:` is the correct PTB v20+ context manager pattern that handles session initialization and cleanup. Instantiating `Bot` outside a context manager and calling `send_message` directly also works but leaves the HTTPX session open — the context manager is cleaner for fire-and-forget.

### Pattern 5: Daily Summary Asyncio Task

**What:** A long-lived coroutine that sleeps until the next UTC midnight, fires the daily summary Telegram message, then sleeps again. Runs as a background `asyncio.create_task` alongside the scan loop.

```python
# Source: asyncio pattern (HIGH confidence — stdlib)
import asyncio
from datetime import datetime, timedelta

async def daily_summary_task(alerter: TelegramAlerter, app_state: AppState):
    while True:
        now = datetime.utcnow()
        tomorrow_midnight = datetime.combine(now.date() + timedelta(days=1), datetime.min.time())
        sleep_seconds = (tomorrow_midnight - now).total_seconds()
        await asyncio.sleep(sleep_seconds)
        # Build and send summary
        text = _build_daily_summary(app_state)
        await alerter.send(text)
```

### Pattern 6: arb_id Threading Through execute_opportunity

**What:** Generate `arb_id = str(uuid.uuid4())` at the top of `execute_opportunity()`, pass it through to both leg results, and write the `arb_pairs` row only when `no_filled == True`. The `arb_id` is returned alongside `ExecutionResult` items so `live_run.py` can pass it to the `arb_pairs` writer.

**Design choice:** Either (a) expand `ExecutionResult` to carry `arb_id`, or (b) return `arb_id` alongside results as a tuple `(arb_id, results)`. Option (b) is cleaner — `ExecutionResult` already has a defined schema and changing it breaks test fixtures.

```python
# In execute_opportunity() — return signature change (HIGH confidence design)
async def execute_opportunity(...) -> tuple[str, list[ExecutionResult]]:
    arb_id = str(uuid.uuid4())
    ...
    return arb_id, results

# In live_run.py:
arb_id, results = await execute_opportunity(client, opp, config, risk_gate, ...)
```

### Pattern 7: arb_pairs Table Write via AsyncWriter Extension

**What:** Add an `enqueue_arb_pair(arb_row: dict)` method to `AsyncWriter` (or create a second queue for arb pairs). The simplest approach: add `insert_arb_pair()` to `schema.py` and call it directly from `live_run.py` after both legs confirm — no need to queue arb_pairs since they're written at most once per executed opportunity.

**When to use:** Direct synchronous SQLite write is acceptable here because `arb_pairs` inserts happen at most a few times per scan cycle, and the existing `conn` is already available in `live_run.py`.

### Anti-Patterns to Avoid

- **Awaiting Telegram in the execution path:** Never `await alerter.send(...)` directly in `execute_opportunity()` — always `asyncio.create_task()`. A Telegram timeout blocks the entire trade.
- **Threading the dashboard:** Never use `threading.Thread` to run uvicorn. It cannot share asyncio objects (Queue, Event, Task) with the main loop safely.
- **Calling `loop.run_until_complete()` inside an already-running loop:** Raises `RuntimeError: This event loop is already running`. Use `asyncio.create_task()` for all concurrent work.
- **SQLite write from multiple threads:** SQLite with `check_same_thread=False` is safe for reads from the FastAPI handler thread, but writes must still happen from the asyncio loop. FastAPI handlers on uvicorn with `loop="none"` run in the same event loop — this is safe.
- **Blocking SQLite query in FastAPI handler:** Queries like `SELECT * FROM trades LIMIT 20` are fast on small datasets (<10k rows). For Phase 4 scale (< $1k capital, < 100 trades/day), direct synchronous SQLite reads in async handlers are acceptable. No `run_in_executor` needed.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Telegram HTTP client | Custom httpx calls to api.telegram.org | `telegram.Bot.send_message()` | PTB handles rate limits, error types, retry-after headers, HTML escaping |
| ASGI server | Custom asyncio HTTP server | uvicorn | Handles HTTP/1.1 keep-alive, header parsing, concurrent request handling; implementing correctly takes weeks |
| HTML escaping for Telegram | `text.replace("<", "&lt;")` etc. | PTB HTML parse mode handles this; use `telegram.helpers.escape_html()` for dynamic content | Missing escapes cause parse errors that silently drop messages |
| UTC midnight calculation | Complex datetime arithmetic | `datetime.combine(date + timedelta(1), time.min) - datetime.utcnow()` | Proven stdlib pattern |
| JSON serialization in FastAPI | `json.dumps(result)` + manual Response | FastAPI's automatic Pydantic/dict serialization | Auto handles datetime, float, None types |

**Key insight:** The Telegram Bot API has many edge cases (HTML entity escaping, message length limits of 4096 chars, rate limiting at 30 msg/s, connection pooling) that PTB handles. Using raw httpx bypasses all of these safeguards.

## Common Pitfalls

### Pitfall 1: uvicorn log_level pollutes Loguru output
**What goes wrong:** uvicorn defaults to `INFO` logging which sends HTTP access logs through Python's stdlib logging, creating noise in Loguru output (which is the project's logging standard).
**Why it happens:** uvicorn uses stdlib `logging` module by default.
**How to avoid:** Set `log_level="warning"` in `uvicorn.Config` to suppress access logs. Errors will still appear. For complete suppression: `access_log=False`.
**Warning signs:** Console floods with `INFO:     127.0.0.1:XXXXX - "GET /api/status HTTP/1.1" 200 OK` every 10 seconds.

### Pitfall 2: Bot() context manager re-instantiation per send
**What goes wrong:** Creating `async with Bot(token):` per message is correct but slightly slower due to HTTPX session setup. For fire-and-forget this is fine, but creates unnecessary HTTP connection overhead if called rapidly.
**Why it happens:** PTB v20+ uses HTTPX under the hood. Each `async with Bot()` creates a new HTTPX client.
**How to avoid:** For Phase 4's alert rate (< 10 alerts/day), per-call context manager is perfectly acceptable. Do not pre-initialize outside the async context (leads to unclosed session warnings).
**Warning signs:** `ResourceWarning: Unclosed client session` in logs.

### Pitfall 3: FastAPI endpoint accessing shared mutable state races
**What goes wrong:** Dashboard endpoint reads `app_state.daily_pnl_usd` while scan loop writes it. In CPython with asyncio (single-threaded), this is safe — context switches only happen at `await` points. But if `uvicorn` is run with `workers=2`, state diverges.
**Why it happens:** Multi-worker uvicorn creates separate processes.
**How to avoid:** Always use `workers=1` (the default for programmatic `Server.serve()`). Never set `workers > 1` in the dashboard config.
**Warning signs:** Dashboard shows stale or inconsistent P&L across refreshes.

### Pitfall 4: execute_opportunity() return type change breaks callers
**What goes wrong:** Changing `execute_opportunity()` to return `tuple[str, list[ExecutionResult]]` breaks `live_run.py`'s unpacking pattern and all existing test mocks.
**Why it happens:** Tests mock `execute_opportunity` with `new_callable=AsyncMock` returning `[MagicMock()]` (a flat list). Changing to a tuple breaks unpacking.
**How to avoid:** Update `test_live_run.py` mock return values to `("arb-uuid", [MagicMock()])` simultaneously with the signature change. Run the test suite after each file change.
**Warning signs:** `ValueError: not enough values to unpack` in test output.

### Pitfall 5: Telegram chat_id type mismatch
**What goes wrong:** `chat_id` passed as a string "123456789" where PTB expects an int. Or vice versa.
**Why it happens:** `.env` file always provides strings; PTB v21+ accepts both str and int for `send_message(chat_id=...)`.
**How to avoid:** Accept `str | None` in `BotConfig.telegram_chat_id`, pass directly to `send_message`. PTB handles coercion internally. Document in secrets.env.example that it's the numeric chat ID as a string.
**Warning signs:** `BadRequest: Chat not found` error from Telegram.

### Pitfall 6: arb_pairs written before NO leg fill_at is set in trades
**What goes wrong:** `arb_pairs.exit_time` computed from `NO leg filled_at` but the trades row hasn't been updated with `filled_at` yet (the Phase 3 `insert_trade()` sets `filled_at=None` and never updates it post-fill).
**Why it happens:** Phase 3 `insert_trade()` captures `submitted_at` at write time but `filled_at` is never updated via an UPDATE statement — it stays NULL.
**How to avoid:** Compute `exit_time = datetime.utcnow().isoformat()` directly in the `arb_pairs` write path (at the moment of confirmed fill), rather than reading it from the `trades` table. The `trades` table's `filled_at` column can be updated as a separate improvement, but `arb_pairs` should not depend on it.
**Warning signs:** `arb_pairs.exit_time = NULL` for all rows.

### Pitfall 7: Docker port 8080 not exposed
**What goes wrong:** Dashboard server starts but is inaccessible from outside the container.
**Why it happens:** `docker-compose.yml` currently has no `ports:` mapping.
**How to avoid:** Add `ports: ["8080:8080"]` to the bot service in `docker-compose.yml`.
**Warning signs:** `curl http://VPS_IP:8080/` times out.

## Open Questions

1. **`insert_trade()` category source**
   - What we know: `live_run.py` calls `insert_trade(conn, result, opp.market_question, trade_id)` — `opp.category` is available on the `ArbitrageOpportunity` object.
   - What's unclear: Whether to add `category` and `config` params to `insert_trade()`, or compute `fees_usd` before calling it.
   - Recommendation: Compute fees in `live_run.py` before calling `insert_trade()`, add `fees_usd: float = 0.0` parameter to `insert_trade()`. Keeps schema.py as a pure persistence layer with no business logic dependency on `BotConfig`.

2. **Dashboard open positions query**
   - What we know: `trades` table has `status IN ('filled', 'partial')` for open positions. `arb_pairs` table has closed arbs.
   - What's unclear: "Open positions" for the dashboard means YES legs filled but NO leg not yet confirmed — essentially trades with `leg='yes' AND status='filled'` that have no corresponding `arb_pairs` row.
   - Recommendation: Query `trades WHERE leg='yes' AND status='filled'` and LEFT JOIN to `arb_pairs` on `yes_trade_id` — rows with no arb_pairs match are open. Keep the query simple for Phase 4; this is a display-only concern.

3. **capital_efficiency metric definition**
   - What we know: D-10 asks for "capital efficiency over time (rolling 7-day, 30-day)".
   - What's unclear: No formula specified. Standard definition is net_pnl / capital_deployed.
   - Recommendation: `capital_efficiency = sum(net_pnl) / total_capital_usd` over the time window, queried from `arb_pairs` filtered by `entry_time`. Can be NULL if no arb_pairs rows exist yet — dashboard displays "N/A".
