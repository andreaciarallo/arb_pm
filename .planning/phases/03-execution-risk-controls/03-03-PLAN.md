Retry-then-hedge sequence (D-03):
```
YES filled → verify REST → attempt NO × 3 retries × 500ms →
  if still NO after 3 → SELL YES at market price=0.01 (hits best bid)
  check risk_gate.is_kill_switch_active() before each NO retry
```

EXEC-04 verification approach (Phase 3 — intentionally REST-only):
```
verify_fill_rest() polls get_order(order_id) every 500ms × 10 iterations = 5s timeout.
User WebSocket fill channel is NOT used in Phase 3.
Reason: RESEARCH.md Pattern 3 rates user channel message format as LOW confidence
(undocumented field names; only URL is confirmed). WebSocket fill primary signal deferred
to Phase 4 when message format can be confirmed against live trades.
This is an intentional design decision, not an omission.
```
